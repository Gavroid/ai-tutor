**Цель:** допустить к закрытому пилоту 4 роли (admin / parent / teacher / student)
без риска получения чужих ответов, потери данных и неработающих дополнительных функций.

**Архитектурный подход:** не «доделать MVP», а **допустить к пилоту именно ту версию,
которая есть**. Убираем всё, что может дать ложные данные; скрываем функции, которые
не доделаны; закрываем самое опасное; перевыпускаем учётки; подключаем внешние страховочные
слои (GitHub, SMB). Ручной deployment остаётся ручным — GitHub Actions внутри домашней сети
в первом этапе не нужен.

**Стек:** существующий; никаких новых зависимостей.

## Принципы первого этапа

1. **Семь пользователей — четыре разные роли.** Внешняя регистрация для пилота не нужна;
   отключаем её и создаём роли заранее.
2. **Достоверность важнее красоты статистики.** Пока правильный ответ и оценка приходят от
   браузера, любой график может быть красиво неправильным.
3. **Скрываем то, что не доделано.** Email worker, PDF-отчёт родителя, admin real-time
   и голосовой ввод не отключаем от кода, но прячем из UI и не используем в пилоте.
4. **Сначала внешние страховочные слои, потом удобства.** Сначала GitHub, SMB, новая учётная
   запись администратора и проверенный deployment-сценарий. Затем — UX, a11y, ручной
   ввод админ-данных.
5. **Семь контейнеров не должны требовать сложной оркестрации.** Один процесс, один хост,
   один deployment-чек-лист — не больше.

## Что делать в первую очередь — почему именно в таком порядке

| Приоритет | Проблема | Что случится, если её не решить |
|---|---|---|
| 1 | `correct_answer` и оценка приходят от браузера | Любые графики и отчёты родителя показывают не то, что реально знает ученик. Это фундамент доверия к системе. |
| 2 | Публичная регистрация позволяет получить роль учителя | Учитель может создавать и публиковать материалы. Для семейной коробки это чужой уровень доступа. |
| 3 | Backup остаётся на той же машине | При потере LXC теряются и работа, и единственная копия. Это превращает пилот в разовую демонстрацию. |
| 4 | Нет внешнего репозитория кода | Невозможно восстановить рабочую версию, если что-то пошло не так; нельзя обсуждать правки с другими специалистами. |
| 5 | Не проверен ручной rollback | Любой deploy становится опасным, потому что нет ответа на вопрос «что делать, если не заработает». |
| 6 | Сырые функции видны в UI | Родитель видит PDF-отчёт, который задублирует 8 запросов и не валидируется; admin real-time возвращает 404; email-уведомления теряются. Всё это создаёт впечатление, что продукт «глючит», даже если четыре роли с базовыми функциями уже работают. |

## Насколько сильно помогут подключение к GitHub и SMB-backup

### Что именно решает каждый из этих слоёв

| Слой | Что защищает | Какой риск снижает | Что **не** покрывает |
|---|---|---|---|
| **GitHub (private repo)** | Исходный код + история изменений | Можно откатить «поломку кода» одной командой `git revert`. История коммитов показывает, кто, когда и почему менял. | Не спасает рабочие данные: ответы ученика, попытки, загруженные файлы. |
| **SMB/NAS backup** | Рабочие данные: БД, файлы, состояние системы | При потере production LXC (аппаратный сбой, случайный `rm -rf`, устаревший Proxmox) можно поднять чистый хост и загрузить вчерашнюю копию. | Не защищает код: если LXC сгорел вместе с workspace, восстановленный код нужно откуда-то взять. |
| **Together** | И код, и данные | Полное восстановление после потери одной машины: GitHub даёт код, SMB даёт данные. | Не даёт онлайн-доступ (нельзя переключиться «на лету»). Не заменяет хороший deployment pipeline. |

### Что это даёт конкретно для проекта

#### GitHub
- **Одна точка правды для кода.** Сейчас код живёт в двух местах: `/root/workspace/ai-tutor`
  и `/opt/ai-tutor` на production. Если что-то потеряется — нет истории и нельзя
  восстановить предыдущую версию. После подключения GitHub: `git log`, `git diff`,
  `git revert` работают.
- **Возможность CI.** GitHub Actions позволяет автоматически прогонять
  `pytest tests/ -q` и `npm run build` на каждом push. Это не даёт уверенности в
  production, но сразу ловит очевидные ошибки.
- **Безопасный обмен с подрядчиками.** Если владелец захочет пригласить внешнего
  ревьюера или копировать примеры — это делается через PR, без передачи
  SSH-доступа к production.
- **Но:** GitHub Actions не подключатся напрямую к `192.168.1.86` (домашняя сеть
  без публичного адреса). Значит, в первом этапе GitHub = история кода + CI,
  а deployment остаётся ручным через `tar | ssh`.

#### SMB/NAS
- **Реальный offsite.** Сейчас offsite-копия остаётся на той же LXC. Если LXC
  умер — пропадёт и рабочая система, и «offsite». SMB-mount на отдельный NAS
  даёт настоящее второе место хранения.
- **Простой протокол.** SMB/CIFS поддерживается любым NAS (Synology, QNAP, TrueNAS,
  обычный Windows-сервер). Никаких специальных клиентов.
- **Восстановление в один шаг.** `rsync -a /mnt/ai-tutor-smb/latest/ /opt/ai-tutor/`
  + `pg_restore` последнего дампа. Не нужны специальные CI-токены или
  GitHub-credentials.
- **Но:** SMB сам по себе не защищает от ransomware (если NAS заражён, бэкапы
  уничтожаются вместе с рабочими данными). Для пилота 4 ролей это не критично;
  для взрослого production — добавится второй offsite.

### Чего GitHub и SMB вместе НЕ решают

- **Автоматический rollback после сломанного deploy.** Для этого нужен
  выделенный CI-runner с доступом в домашнюю сеть, что выходит за рамки
  пилотного этапа.
- **Защиту от ransomware на production.** Если упадёт основная система,
  SMB-копии могут быть поражены. В первом этапе считаем это неприоритетным.
- **Канареечный deploy и мониторинг ошибок в реальном времени.** Это задачи
  следующих этапов.

### Итого по эффекту

| Что хотим | GitHub | SMB | Оба |
|---|:---:|:---:|:---:|
| Защита кода (история, ревью, CI) | ✅ | ❌ | ✅ |
| Защита рабочих данных (БД, файлы) | ❌ | ✅ | ✅ |
| Возможность восстановления после потери LXC | ❌ | частично (только данные) | ✅ |
| CI без ручного вмешательства | ✅ | ❌ | ✅ |
| Непрерывность при сбое production | ❌ | ❌ | ❌ (нужен второй LXC) |
| Безопасный обмен с подрядчиком | ✅ | ❌ | ✅ |

**Вывод: подключение GitHub и SMB вместе закрывает 80–90% типовых рисков для
семейного пилота. Эти два слоя — наименьший срок за наибольший эффект. Их
отсутствие означает, что любой сбой превращает пилот в разовую демонстрацию;
их наличие означает, что даже серьёзный инцидент переживается за часы.**

Ориентировочные трудозатраты: 6–10 часов на подключение GitHub (создать repo,
настроить CI, безопасно отправить код), 4–6 часов на SMB (mount, замена offsite
скрипта, проверка восстановления). Итого **10–16 часов** внутри фазы 4.
